# Продвинутые стратегии ретривера

## Проблема базового векторного поиска

Эмбеддинг запроса пользователя и эмбеддинг нужного документа могут не совпасть из-за различий в терминологии или контексте — даже если документ семантически релевантен. Это называется **mismatch проблемой**.

$$
\text{sim}(\text{embed}(q),\ \text{embed}(d)) < \theta \quad \text{- relevant document skipped}
$$

## Стратегия 1: Step-Back Prompting (промптинг с отступом)

Техника переформулировки запроса: конкретный вопрос → более общий вопрос для поиска.

**Пример:**
```
Исходный: "В какой команде играл Тьерри Одель с 2007 по 2008 год?"
Step-back: "Какова карьерная история Тьерри Оделя?"
```

Более широкий запрос охватывает больший диапазон документов → вероятность нахождения нужного растёт.

**Полный конвейер с step-back:**

$$
q_\text{step} = \text{LLM}(\text{step\_back\_prompt},\ q_\text{original})
$$
$$
\text{docs} = \text{VectorSearch}(q_\text{step})
$$
$$
\text{answer} = \text{LLM}(q_\text{original},\ \text{docs})
$$

Поиск выполняется по переформулированному вопросу, но ответ генерируется на исходный.

**Промптинг**: используется **few-shot** (2–5 примеров) для стабильных результатов. Zero-shot тоже работает, но менее предсказуемо.

## Стратегия 2: Ретривер родительских документов

**Идея**: встраивать мелкие чанки (для точного поиска), возвращать крупный родительский документ (для полного контекста).

```
PDF
 └── Parent (2000 символов) ──HAS_PARENT──► PDF
      ├── Child (500 символов) [embedding]
      ├── Child (500 символов) [embedding]
      └── Child (500 символов) [embedding]
```

**Почему это работает**: при встраивании длинного документа вектор усредняет все идеи → конкретный запрос хуже совпадает. Мелкий чанк точнее представляет одну мысль.

**Нюанс поиска**: при векторном поиске берут `k × 4` дочерних узлов, затем дедуплицируют родительские → финально возвращают `k` уникальных родителей.

$$
\text{initial\_pool} = \text{VectorSearch}(q,\ k \times 4)
$$
$$
\text{result} = \text{TopK}\!\left(\text{deduplicate\_parents}(\text{initial\_pool}),\ k\right)
$$

## Стратегия 3: Гипотетические вопросы (Hypothetical Question Embedding)

Вместо встраивания самого документа — встраивание **вопросов, на которые он может ответить**:

```
Документ: "Леопольд получила степень магистра ботаники в Беркли в 1950 году."
Гипотетический вопрос: "Что изучала Леопольд в Калифорнийском университете?"
```

Запрос пользователя совпадёт с вопросом лучше, чем с сырым текстом документа.

## Другие техники повышения точности

| Техника | Суть | Применение |
|---------|------|-----------|
| **Переранжирование (Reranking)** | Второй проход с более сложной моделью для переупорядочивания результатов | После первичного поиска |
| **Фильтрация по метаданным** | Ограничение кандидатов по дате, автору, типу | Перед семантическим поиском |
| **Дообучение модели эмбеддингов** | Адаптация к предметной области | Требует инфраструктуры и пересчёта всех эмбеддингов |
| **Гибридный поиск** | Вектор + BM25 (ключевые слова) | Запросы с конкретными именами, терминами |

## Гибридный поиск: нормализация оценок

Оценки векторного и полнотекстового поиска разные по шкале → нужна нормализация:

$$
\text{score\_norm}(d) = \frac{\text{score}(d)}{\max_i \text{score}(d_i)}
$$

Финальный результат: объединение двух списков с дедупликацией, сортировка по максимальной нормализованной оценке.

**Ограничение гибридного поиска**: работает хорошо для точных совпадений (имена, термины, аббревиатуры), но общий уровень качества всё ещё ограничен неструктурированными данными — для агрегации и фильтрации нужны структурированные данные.
